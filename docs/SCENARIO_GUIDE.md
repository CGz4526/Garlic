# 从业务场景理解技术用途

> 本文换个角度讲技术：不按"技术栈"罗列实现，而是按"项目要解决的业务场景"组织——每个场景抛出"遇到了什么问题"，再说"哪个技术来解决"，最后点明"它在本项目里具体服务哪个功能"。
>
> 阅读群体：已经读过 [架构设计](ARCHITECTURE.md) / [学习笔记](STUDY_NOTES.md)，了解实现细节，但想弄清楚每个技术到底"为什么存在、给谁服务"。

---

## 一、先分清两类需求

项目里的技术分两种用途：

- **业务功能型**：用户能直接感知的功能（点短链跳转、看点击统计、创建短链）
- **底层支撑型**：用户感知不到，但系统跑起来必须有的能力（数据存得下、扛得住高压、出问题能发现）

下面按 8 个场景拆，每个场景点明它服务的是"功能"还是"支撑"。

---

## 场景 A：用户点短链要"秒开"（业务功能）

**问题**：每次点短链都去数据库查原始长链，10 毫秒一次。一个人点没事，10 万人同时点，数据库直接卡死，用户看到的是转圈圈。

**解决：三级缓存 Caffeine → Redis → MySQL**

| 层 | 技术 | 在本项目干什么 |
|----|------|--------------|
| L1 | Caffeine | 本地内存缓存，最近查过的短链直接命中，0.05ms。服务「短链跳转」功能——绝大多数重复点击走这层 |
| L2 | Redis | 分布式缓存，L1 没有就查它，1ms。服务「短链跳转」+「短链详情查询」 |
| L3 | MySQL | 兜底数据库，10ms。只有两层缓存都没有才查它 |

**配套技术**：

- **布隆过滤器**（Redisson 实现）：门卫角色。有人点一个根本不存在的短码，先在这里判断"肯定不存在"，直接返回 404，不往下查。服务「短链跳转」防缓存穿透。
- **Redisson 分布式锁**：L1/L2 都没有、高并发同时来查同一个短码时，只放一个请求去查数据库回填缓存，其他请求等。服务「短链跳转」防缓存击穿。

> 一句话：这套东西全是给「点短链跳转」这一个功能服务的，目标是让 99% 的点击在 1ms 内返回。

## 场景 B：明星发微博瞬间 10 万人点同一条短链（底层支撑）

**问题**：瞬时流量是平时的 100 倍，服务器扛不住会崩；还有人写脚本恶意刷，会把接口打挂。

**解决：多级限流 + 熔断 + 黑名单**

| 技术 | 在本项目干什么 |
|------|--------------|
| Redis + Lua 令牌桶 | 接口级限流，全局每秒最多 5000 次点击通过。服务**所有接口**的总体防护 |
| Redis ZSet 滑动窗口 | 用户级/IP 级限流，单用户每秒最多点 50 次。服务「创建短链」「短链跳转」防一人刷爆 |
| Sentinel | 熔断降级。当跳转接口响应慢到 100ms 以上占比超 50%，自动"跳闸"几秒，期间直接返回 Caffeine 本地缓存的老数据。服务「短链跳转」的高可用保命 |
| IP 黑名单（自研） | 分钟级计数器，某 IP 一分钟点超 1000 次自动封禁 1 小时。服务「短链跳转」防恶意刷量 |

> 一句话：这套是给「系统不被冲垮」这个底层目标服务的，不直接对应某个业务功能，但保所有功能活着。

## 场景 C：点击统计不能拖垮跳转主流程（业务功能：访问统计）

**问题**：每次点短链都要记一笔"谁点的、什么时候、用什么手机"。如果每点一次就往数据库写一次，10 万人同时点 = 10 万次数据库写入，数据库崩了，连跳转也跟着崩。

**解决：Kafka 削峰 + 异步聚合**

这条链路是给「访问统计」功能服务的，链路每一步的作用：

```
用户点短链
   ↓
[1] 本地队列(10000容量)   ← 削峰：先把点击事件攒在内存队列，不直接写库
   ↓
[2] Kafka                  ← 解耦+缓冲：把点击事件转发给后台统计消费者，跳转主流程到此就结束了（用户已经跳走）
   ↓
[3] 消费者内存聚合          ← 攒批：不是来一条写一条，而是在内存里按短码+小时归类累加
   ↓
[4] 定时器 60s 落库         ← 批量写：每 60 秒把算好的统计结果一次性写进数据库
```

**为什么需要 Kafka**（这正是核心疑问）：

- 如果没有 Kafka，点击事件要么直接写库（数据库崩），要么堆在本地内存（服务重启就丢、多实例间无法共享）。
- Kafka 在这里扮演**"缓冲水池"**：跳转服务只管把事件扔进 Kafka 就完事（快），统计消费者按自己的节奏慢慢消费。即使统计消费者挂了，事件也存在 Kafka 里不丢；即使瞬时 10 万点击，Kafka 也能先攒着不冲垮消费者。
- **它给「访问统计」功能服务**，解决"高并发下点击数据既要不丢、又不能拖垮跳转主链路"的场景。

**配套技术**：

- **Redis HyperLogLog**：UV（独立访客数）去重。10 万人点同一短链，UV 应该是 10 万而不是 100 万次点击。用普通 Set 存 10 万个用户标识要几 MB，HyperLogLog 只要 12KB，误差 0.81%。服务「统计总览」功能的 UV 计算。
- **线程池隔离**（statsExecutor 等 4 个）：统计用的线程池和跳转用的分开，统计慢了不影响跳转。服务系统隔离。

> 一句话：Kafka 这条链路是给「访问统计」功能服务的，解决"统计不能拖垮跳转"的场景。

## 场景 D：短链数据越积越多，单表查不动（底层支撑）

**问题**：短链服务跑半年，短链表存了 1 亿条，单表查询从 10ms 变成 2 秒，统计表更夸张（每次点击都记一条），一个月就上亿条。

**解决：ShardingSphere 分库分表**

| 技术 | 在本项目干什么 |
|------|--------------|
| ShardingSphere-JDBC | 把短链表拆成 4 库 × 4 表 = 16 片，数据分散存。服务**所有短链查询功能**，让单表数据量从 1 亿降到 600 万，查询恢复秒级 |
| INLINE 分片算法 | 按 short_code 哈希路由到具体分片。服务「按短码查询」（跳转/详情），保证点同一个短码总查同一张表 |
| INTERVAL 月份分表 | 访问明细表按 access_time 月份分表（202601、202602...）。服务「访问明细」归档，老数据可以直接整表删 |
| 绑定表 | 短链表和统计表用同一分片键，join 时不跨库。服务「统计查询」性能 |
| 广播表 | t_user/t_group 每个库存一份，避免跨库 join。服务「创建短链」时的用户校验 |
| 雪花算法 + Base62 | 生成全局唯一 ID + 6~7 位短码。服务「创建短链」功能，分库分表下 ID 不能重复 |

> 一句话：这套是给「数据存得下、查得动」这个底层目标服务的，对应业务上是「创建/查询/统计」所有功能的数据基础。

## 场景 E：改了短链信息，所有服务器的缓存要同步（业务功能：短链管理）

**问题**：用户改了短链的描述，但 Caffeine 本地缓存还在 100 台服务器上各存了一份旧的。用户查询时可能命中旧缓存，看到旧描述。

**解决：缓存一致性方案（延迟双删 + Kafka 广播）**

| 技术 | 在本项目干什么 |
|------|--------------|
| 延迟双删 | 更新数据库前删一次缓存，更新后延迟 500ms 再删一次。服务「更新短链」功能，覆盖主从复制延迟窗口 |
| Kafka 广播 | 更新后发一条失效消息到 Kafka，所有实例收到后清自己的 Caffeine。服务「更新/删除短链」的多实例缓存同步 |
| TTL 随机抖动 | 缓存过期时间设成 3600+随机(0,300)秒。服务**防雪崩**，避免同一时刻大量缓存同时失效全打向数据库 |

> 一句话：给「短链管理（改/删）」功能服务，解决"多实例本地缓存如何保持一致"的场景。

## 场景 F：创建短链不能重复、批量要快（业务功能：创建/批量创建）

**问题**：用户提交同一个长链两次，不能生成两个不同短码（幂等）；批量提交 100 个长链，串行处理要 1 秒，太慢。

**解决**：

| 技术 | 在本项目干什么 |
|------|--------------|
| Redis SETNX | 创建前先查"这个长链是不是建过了"。服务「创建短链」的幂等 |
| Redisson 分布式锁 | SETNX 没查到时，加分布式锁再去数据库查一次（双重检查），防止并发下两个请求都通过 SETNX 检查。服务「创建短链」幂等的并发安全 |
| CompletableFuture + batchGen 线程池 | 批量创建时 100 个长链并发处理，单条失败不影响其他。服务「批量创建」功能的性能 |

> 一句话：给「创建/批量创建短链」功能服务，解决"幂等不重复 + 批量快"的场景。

## 场景 G：系统跑起来要能看见、出问题要能发现（底层支撑）

**问题**：线上跑着，限流触发了多少次？哪个线程池拒绝过任务？GC 耗时多少？没有这些数据，出了问题只能瞎猜。

**解决：监控体系**

| 技术 | 在本项目干什么 |
|------|--------------|
| Spring Boot Actuator | 暴露 /actuator/health 等端点。服务**健康检查**（MySQL/Redis 是否连得上） |
| Micrometer | 应用内指标采集（线程池拒绝数、缓存命中率）。服务**业务指标可观测** |
| Prometheus | 抓取 Micrometer 指标存起来供查询。服务**指标存储与查询** |
| 全局异常处理（@RestControllerAdvice） | 所有接口异常统一包装成错误码返回。服务**所有接口**的错误一致性 |

> 一句话：给「运维可观测」这个底层目标服务，不对应单一业务功能。

## 附：功能驱动的简历描述

> 与 [RESUME.md](RESUME.md) 的「技术：具体实现」视角不同，下面按「功能：用什么技术思路」组织，专注设计思路而非代码细节，适合用于面试讲述或简历功能段。

**Garlic 高并发短链接服务** 2026年05月 - 2026年07月

**短链跳转秒开**：设计 Caffeine 本地缓存 → Redis 分布式缓存 → MySQL 三级查询链路，让 99% 的点击在 1ms 内返回；布隆过滤器在入口拦截不存在的短码防穿透，分布式锁控制并发回填缓存防击穿，TTL 随机抖动防雪崩，保证高压下跳转接口稳定在 P99 < 20ms。

**访问统计不拖垮主链路**：把"记一笔点击"从跳转主流程剥离——点击事件先进本地内存队列削峰，再丢 Kafka 做缓冲水池，后台消费者按短码+小时在内存聚合，定时器每 60 秒批量落库；UV 用 Redis HyperLogLog 去重（12KB 内存扛 10 万级访客），既不丢数据又不冲垮数据库，统计与跳转物理隔离互不影响。

**高并发防冲垮**：四层防护应对瞬时流量——Redis+Lua 令牌桶做接口级限流（5000 QPS），ZSet 滑动窗口做用户级/IP 级限流（50 次/秒），Sentinel 慢调用比例熔断后降级读本地缓存保可用，IP 黑名单按分钟级计数动态封禁恶意流量，让系统扛住瞬时 10 万点击不崩。

**短链创建幂等不重复**：Redis SETNX 快速判重 + Redisson 分布式锁双重检查，保证同一长链无论被并发提交多少次只生成一个短码；批量创建用 CompletableFuture 并发编排 + 独立线程池隔离，单条失败不影响其他，100 条批量耗时接近单条。

**短链改删的缓存一致性**：采用延迟双删（更新前后各删一次，间隔 500ms 覆盖主从复制延迟）+ Kafka 广播失效消息（通知所有实例清本地 Caffeine）+ TTL 兜底，三重保证多实例部署下本地缓存与数据库最终一致。

**数据存储水平扩容**：用 ShardingSphere 把短链表拆成 4 库 × 4 表 = 16 分片（按 short_code 哈希路由），访问明细表按月份自动分表便于归档，统计表与短链表绑定同分片键避免跨库 join，用户/分组表广播到每个库避免跨库查询；雪花算法生成全局唯一 ID + Base62 编码产出 6~7 位短码，分库分表下 ID 不冲突、容量达 568 亿。

**运维可观测**：Actuator 暴露健康检查（MySQL/Redis 连通性），Micrometer 采集线程池拒绝数、缓存命中率、限流触发次数等业务指标并对接 Prometheus，全局异常处理器统一错误码返回，让线上问题可发现、可定位。

**JVM 调优支撑高压**：G1 收集器 + 2G 固定堆 + MaxGCPauseMillis=100ms 目标停顿，单节点稳定支撑 5000 QPS，P99 < 20ms。

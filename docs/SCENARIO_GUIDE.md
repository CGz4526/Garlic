## 一、先分清两类需求

项目里的技术分两种用途：

- **业务功能型**：用户能直接感知的功能（点短链跳转、看点击统计、创建短链）
- **底层支撑型**：用户感知不到，但系统跑起来必须有的能力（数据存得下、扛得住高压、出问题能发现）

下面按 8 个场景拆，每个场景点明它服务的是"功能"还是"支撑"。

---

## 场景 A：用户点短链要"秒开"（业务功能）

**问题**：每次点短链都去数据库查原始长链，10 毫秒一次。一个人点没事，10 万人同时点，数据库直接卡死，用户看到的是转圈圈。

**解决：三级缓存 Caffeine → Redis → MySQL**

| 层 | 技术 | 在本项目干什么 |
|----|------|--------------|
| L1 | Caffeine | 本地内存缓存，最近查过的短链直接命中，0.05ms。服务「短链跳转」功能——绝大多数重复点击走这层 |
| L2 | Redis | 分布式缓存，L1 没有就查它，1ms。服务「短链跳转」+「短链详情查询」 |
| L3 | MySQL | 兜底数据库，10ms。只有两层缓存都没有才查它 |

**配套技术**：

- **布隆过滤器**（Redisson 实现）：门卫角色。有人点一个根本不存在的短码，先在这里判断"肯定不存在"，直接返回 404，不往下查。服务「短链跳转」防缓存穿透。
- **Redisson 分布式锁**：L1/L2 都没有、高并发同时来查同一个短码时，只放一个请求去查数据库回填缓存，其他请求等。服务「短链跳转」防缓存击穿。

> 一句话：这套东西全是给「点短链跳转」这一个功能服务的，目标是让 99% 的点击在 1ms 内返回。

## 场景 B：明星发微博瞬间 10 万人点同一条短链（底层支撑）

**问题**：瞬时流量是平时的 100 倍，服务器扛不住会崩；还有人写脚本恶意刷，会把接口打挂。

**解决：多级限流 + 熔断 + 黑名单**

| 技术 | 在本项目干什么 |
|------|--------------|
| Redis + Lua 令牌桶 | 接口级限流，全局每秒最多 5000 次点击通过。服务**所有接口**的总体防护 |
| Redis ZSet 滑动窗口 | 用户级/IP 级限流，单用户每秒最多点 50 次。服务「创建短链」「短链跳转」防一人刷爆 |
| Sentinel | 熔断降级。当跳转接口响应慢到 100ms 以上占比超 50%，自动"跳闸"几秒，期间直接返回 Caffeine 本地缓存的老数据。服务「短链跳转」的高可用保命 |
| IP 黑名单（自研） | 分钟级计数器，某 IP 一分钟点超 1000 次自动封禁 1 小时。服务「短链跳转」防恶意刷量 |

> 一句话：这套是给「系统不被冲垮」这个底层目标服务的，不直接对应某个业务功能，但保所有功能活着。

## 场景 C：点击统计不能拖垮跳转主流程（业务功能：访问统计）

**问题**：每次点短链都要记一笔"谁点的、什么时候、用什么手机"。如果每点一次就往数据库写一次，10 万人同时点 = 10 万次数据库写入，数据库崩了，连跳转也跟着崩。

**解决：Kafka 削峰 + 异步聚合**

这条链路是给「访问统计」功能服务的，链路每一步的作用：

```
用户点短链
   ↓
[1] 本地队列(10000容量)   ← 削峰：先把点击事件攒在内存队列，不直接写库
   ↓
[2] Kafka                  ← 解耦+缓冲：把点击事件转发给后台统计消费者，跳转主流程到此就结束了（用户已经跳走）
   ↓
[3] 消费者内存聚合          ← 攒批：不是来一条写一条，而是在内存里按短码+小时归类累加
   ↓
[4] 定时器 60s 落库         ← 批量写：每 60 秒把算好的统计结果一次性写进数据库
```

**为什么需要 Kafka**（这正是核心疑问）：

- 如果没有 Kafka，点击事件要么直接写库（数据库崩），要么堆在本地内存（服务重启就丢、多实例间无法共享）。
- Kafka 在这里扮演**"缓冲水池"**：跳转服务只管把事件扔进 Kafka 就完事（快），统计消费者按自己的节奏慢慢消费。即使统计消费者挂了，事件也存在 Kafka 里不丢；即使瞬时 10 万点击，Kafka 也能先攒着不冲垮消费者。
- **它给「访问统计」功能服务**，解决"高并发下点击数据既要不丢、又不能拖垮跳转主链路"的场景。

**配套技术**：

- **Redis HyperLogLog**：UV（独立访客数）去重。10 万人点同一短链，UV 应该是 10 万而不是 100 万次点击。用普通 Set 存 10 万个用户标识要几 MB，HyperLogLog 只要 12KB，误差 0.81%。服务「统计总览」功能的 UV 计算。
- **线程池隔离**（statsExecutor 等 4 个）：统计用的线程池和跳转用的分开，统计慢了不影响跳转。服务系统隔离。

> 一句话：Kafka 这条链路是给「访问统计」功能服务的，解决"统计不能拖垮跳转"的场景。

## 场景 D：短链数据越积越多，单表查不动（底层支撑）

**问题**：短链服务跑半年，短链表存了 1 亿条，单表查询从 10ms 变成 2 秒，统计表更夸张（每次点击都记一条），一个月就上亿条。

**解决：ShardingSphere 分库分表**

| 技术 | 在本项目干什么 |
|------|--------------|
| ShardingSphere-JDBC | 把短链表拆成 4 库 × 4 表 = 16 片，数据分散存。服务**所有短链查询功能**，让单表数据量从 1 亿降到 600 万，查询恢复秒级 |
| INLINE 分片算法 | 按 short_code 哈希路由到具体分片。服务「按短码查询」（跳转/详情），保证点同一个短码总查同一张表 |
| INTERVAL 月份分表 | 访问明细表按 access_time 月份分表（202601、202602...）。服务「访问明细」归档，老数据可以直接整表删 |
| 绑定表 | 短链表和统计表用同一分片键，join 时不跨库。服务「统计查询」性能 |
| 广播表 | t_user/t_group 每个库存一份，避免跨库 join。服务「创建短链」时的用户校验 |
| 雪花算法 + Base62 | 生成全局唯一 ID + 6~7 位短码。服务「创建短链」功能，分库分表下 ID 不能重复 |

> 一句话：这套是给「数据存得下、查得动」这个底层目标服务的，对应业务上是「创建/查询/统计」所有功能的数据基础。

## 场景 E：改了短链信息，所有服务器的缓存要同步（业务功能：短链管理）

**问题**：用户改了短链的描述，但 Caffeine 本地缓存还在 100 台服务器上各存了一份旧的。用户查询时可能命中旧缓存，看到旧描述。

**解决：缓存一致性方案（延迟双删 + Kafka 广播）**

| 技术 | 在本项目干什么 |
|------|--------------|
| 延迟双删 | 更新数据库前删一次缓存，更新后延迟 500ms 再删一次。服务「更新短链」功能，覆盖主从复制延迟窗口 |
| Kafka 广播 | 更新后发一条失效消息到 Kafka，所有实例收到后清自己的 Caffeine。服务「更新/删除短链」的多实例缓存同步 |
| TTL 随机抖动 | 缓存过期时间设成 3600+随机(0,300)秒。服务**防雪崩**，避免同一时刻大量缓存同时失效全打向数据库 |

> 一句话：给「短链管理（改/删）」功能服务，解决"多实例本地缓存如何保持一致"的场景。

## 场景 F：创建短链不能重复、批量要快（业务功能：创建/批量创建）

**问题**：用户提交同一个长链两次，不能生成两个不同短码（幂等）；批量提交 100 个长链，串行处理要 1 秒，太慢。

**解决**：

| 技术 | 在本项目干什么 |
|------|--------------|
| Redis SETNX | 创建前先查"这个长链是不是建过了"。服务「创建短链」的幂等 |
| Redisson 分布式锁 | SETNX 没查到时，加分布式锁再去数据库查一次（双重检查），防止并发下两个请求都通过 SETNX 检查。服务「创建短链」幂等的并发安全 |
| CompletableFuture + batchGen 线程池 | 批量创建时 100 个长链并发处理，单条失败不影响其他。服务「批量创建」功能的性能 |

> 一句话：给「创建/批量创建短链」功能服务，解决"幂等不重复 + 批量快"的场景。

## 场景 G：系统跑起来要能看见、出问题要能发现（底层支撑）

**问题**：线上跑着，限流触发了多少次？哪个线程池拒绝过任务？GC 耗时多少？没有这些数据，出了问题只能瞎猜。

**解决：监控体系**

| 技术 | 在本项目干什么 |
|------|--------------|
| Spring Boot Actuator | 暴露 /actuator/health 等端点。服务**健康检查**（MySQL/Redis 是否连得上） |
| Micrometer | 应用内指标采集（线程池拒绝数、缓存命中率）。服务**业务指标可观测** |
| Prometheus | 抓取 Micrometer 指标存起来供查询。服务**指标存储与查询** |
| 全局异常处理（@RestControllerAdvice） | 所有接口异常统一包装成错误码返回。服务**所有接口**的错误一致性 |

> 一句话：给「运维可观测」这个底层目标服务，不对应单一业务功能。

## 附：功能驱动的简历描述

> 与 [RESUME.md](RESUME.md) 的「技术：具体实现」视角不同，下面按「功能：用什么技术思路」组织，专注设计思路而非代码细节，适合用于面试讲述或简历功能段。

**Garlic 高并发短链接服务** 2026年05月 - 2026年07月

**短链跳转秒开**：设计 Caffeine 本地缓存 → Redis 分布式缓存 → MySQL 三级查询链路，让 99% 的点击在 1ms 内返回；布隆过滤器在入口拦截不存在的短码防穿透，分布式锁控制并发回填缓存防击穿，TTL 随机抖动防雪崩，保证高压下跳转接口稳定在 P99 < 20ms。

**访问统计不拖垮主链路**：把"记一笔点击"从跳转主流程剥离——点击事件先进本地内存队列削峰，再丢 Kafka 做缓冲水池，后台消费者按短码+小时在内存聚合，定时器每 60 秒批量落库；UV 用 Redis HyperLogLog 去重（12KB 内存扛 10 万级访客），既不丢数据又不冲垮数据库，统计与跳转物理隔离互不影响。

**高并发防冲垮**：四层防护应对瞬时流量——Redis+Lua 令牌桶做接口级限流（5000 QPS），ZSet 滑动窗口做用户级/IP 级限流（50 次/秒），Sentinel 慢调用比例熔断后降级读本地缓存保可用，IP 黑名单按分钟级计数动态封禁恶意流量，让系统扛住瞬时 10 万点击不崩。

**短链创建幂等不重复**：Redis SETNX 快速判重 + Redisson 分布式锁双重检查，保证同一长链无论被并发提交多少次只生成一个短码；批量创建用 CompletableFuture 并发编排 + 独立线程池隔离，单条失败不影响其他，100 条批量耗时接近单条。

**短链改删的缓存一致性**：采用延迟双删（更新前后各删一次，间隔 500ms 覆盖主从复制延迟）+ Kafka 广播失效消息（通知所有实例清本地 Caffeine）+ TTL 兜底，三重保证多实例部署下本地缓存与数据库最终一致。

**数据存储水平扩容**：用 ShardingSphere 把短链表拆成 4 库 × 4 表 = 16 分片（按 short_code 哈希路由），访问明细表按月份自动分表便于归档，统计表与短链表绑定同分片键避免跨库 join，用户/分组表广播到每个库避免跨库查询；雪花算法生成全局唯一 ID + Base62 编码产出 6~7 位短码，分库分表下 ID 不冲突、容量达 568 亿。

**运维可观测**：Actuator 暴露健康检查（MySQL/Redis 连通性），Micrometer 采集线程池拒绝数、缓存命中率、限流触发次数等业务指标并对接 Prometheus，全局异常处理器统一错误码返回，让线上问题可发现、可定位。

**JVM 调优支撑高压**：G1 收集器 + 2G 固定堆 + MaxGCPauseMillis=100ms 目标停顿，单节点稳定支撑 5000 QPS，P99 < 20ms。



### 深入理解：四层防护的技术选型与原理

#### 四层的协同关系（漏斗式逐层收紧）

```
请求进来
   ↓
[第1层] IP 黑名单 —— 进名单的直接拒（最外层，最快）
   ↓
[第2层] 令牌桶 —— 全局总闸，超 5000 QPS 拒
   ↓
[第3层] 滑动窗口 —— 单用户限流，超 50 次/秒拒
   ↓
[第4层] Sentinel 熔断 —— 前三层都过了，但下游变慢时主动降级保命
   ↓
正常处理
```

为什么需要四层而不是一层：

- 只有令牌桶：单个用户能在 5000 配额内刷爆某个接口
- 只有滑动窗口：总流量没控住，百万用户各点 50 次 = 500 万 QPS，系统照样崩
- 只有黑名单：黑名单生效前（1 分钟内）已经造成损害
- 只有熔断：熔断是事后补救，不能替代事前限流

四层各管一段：黑名单挡恶意、令牌桶控总量、滑动窗控单点、熔断保底线。缺任何一层都有漏洞。

---

#### 1. Redis + Lua 令牌桶：为什么这么选

**令牌桶在干什么**：桶里以固定速率滴令牌（每秒 5000 个），请求来时从桶里拿一个；桶里有令牌则通过，没令牌则拒绝。桶有容量上限（如 6000）防止令牌无限积攒。

**为什么用令牌桶而不是漏桶**：

| 算法   | 特点                                       | 问题                                   |
| ------ | ------------------------------------------ | -------------------------------------- |
| 漏桶   | 严格匀速出水（每秒 5000 次，多一点都不行） | 突发流量直接拒绝，即使系统其实扛得住   |
| 令牌桶 | 允许突发——桶里攒的令牌可以瞬间消费         | 符合互联网流量特征：平时平稳，偶尔尖峰 |

互联网流量不是匀速的，用户点击有突发性（比如刚发出去短链，前 100ms 大家集体点）。令牌桶允许"平时攒令牌、尖峰时一次消费"，更贴合实际。

**为什么用 Redis 而不是单机限流**：单机限流（如 Guava RateLimiter）只在单个 JVM 内计数。本项目多实例部署，4 台机器各自限流 5000 QPS，实际总流量能到 20000 QPS，下游直接被打爆。Redis 是全局共享的，4 台机器都从同一个 Redis 拿令牌，总流量被精准控在 5000。

**为什么用 Lua 脚本（关键）**：令牌桶算法本质是三步——查当前令牌数+上次补充时间 → 计算该补充多少 → 判断够不够+扣减+更新。这三步必须原子，否则高并发下多个请求同时查到"令牌够"，同时扣减，超卖。

| 方案                    | 问题                                                     |
| ----------------------- | -------------------------------------------------------- |
| 普通命令                | 三步分开执行，中间能被其他请求插入，超卖                 |
| MULTI/EXEC 事务 + WATCH | 乐观锁，高并发下大量重试，性能差                         |
| Lua 脚本                | Redis 单线程执行 Lua，中间不会被打断，一次完成，性能最优 |

> Redis 单线程 + Lua 原子性 = 高并发下限流计数不超卖，且性能接近原生 Redis 命令。

#### 2. Redis ZSet 滑动窗口：核心原理

##### Q1："限流周期"指什么？是令牌桶吗？

**不是令牌桶。** 令牌桶是另一种限流算法。这里讲的是"窗口类"限流算法。

"限流周期"指的是**限流的时间窗口大小**：

- "50 次/秒" → 周期是 1 秒
- "1000 次/分钟" → 周期是 1 分钟
- "100 次/100ms" → 周期是 100 毫秒

就是"在多长时间内允许通过多少次"里的那个时间长度。

**两种窗口的区别**：

固定窗口：窗口的起止时间是**预先钉死**的，对齐到整时间刻度。

```
窗口1: [12:00:00.000 ─────── 12:00:01.000)  窗口2: [12:00:01.000 ─────── 12:00:02.000)
                                         ↑
                                   钉死的边界，不会动
```

限流 50 次/秒时：

- 12:00:00.999 时点了 50 次 → 窗口1 满，拒绝
- 12:00:01.001 时 → 新窗口2 开始，计数清零，又能点 50 次
- **2 毫秒内通过了 100 次**，是阈值的 2 倍

这就是边界突刺：在窗口切换的瞬间，流量能翻倍。

滑动窗口：窗口的起止时间**随当前时间动态移动**，不对齐到整刻度。

```
当前时刻 12:00:00.999，窗口 = [12:00:00.000 ─── 12:00:00.999)  往前推 1 秒
当前时刻 12:00:01.001，窗口 = [12:00:00.002 ─── 12:00:01.001)  往前推 1 秒
```

12:00:00.999 时点了 50 次，12:00:01.001 时再点，窗口已经滑过去，那 50 次还在窗口内，计数还是 50，第 51 次被拒。没有边界突刺。

> 一句话：固定窗口像"每秒整点清零计数器"，滑动窗口像"永远只看最近 1 秒内有多少次"。

##### Q2：ZSet 实现滑动窗口——时间戳排序？窗口大小怎么维护？匀速滑动吗？

**用时间戳作为 score 排序：是的。** ZSet 里每个元素的结构：

```
key    = rate:user:123
member = 请求唯一标识（用 UUID 或时间戳+随机数，防止同毫秒请求覆盖）
score  = 这次请求的时间戳（毫秒）
```

ZSet 自动按 score（时间戳）从小到大排序，最早请求排在最前面。

**不是匀速滑动，是请求驱动。** 这是关键——滑动窗口的"滑动"是请求驱动的，不是时间驱动的。没有请求时窗口不动；有请求来时，窗口的右端点瞬间跳到"当前时间戳"。

每次请求来时做三步：

```
请求来，当前时间戳 = now（比如 1700000000500 毫秒）
窗口大小 = 1000ms

第1步: ZREMRANGEBYSCORE rate:user:123 0 (now - 1000)
       ↑ 删除 score < (now-1000) 的旧请求
       ↑ 即删除"1 秒前"的所有请求
       ↑ 这一步就是在"滑动窗口的左边界"

第2步: ZCARD rate:user:123
       ↑ 数剩下多少个元素
       ↑ 这就是"当前窗口内（最近1秒）的请求数"

第3步: 如果 < 50 → ZADD rate:user:123 now uuid  （把这次请求记进去）
       如果 >= 50 → 拒绝
```

**图解滑动过程**（限流 3 次/秒，窗口 1000ms）：

```
时刻 t=100:  请求A来
  删除 score<100-1000=0（负数，无）  → ZSet: {A:100}
  ZCARD=1 < 3 → 放行 → ZSet: {A:100}

时刻 t=300:  请求B来
  删除 score<300-1000=0（负数，无） → ZSet: {A:100, B:300}
  ZCARD=2 < 3 → 放行 → ZSet: {A:100, B:300}

时刻 t=500:  请求C来
  删除 score<500-1000=0（负数，无） → ZSet: {A:100, B:300, C:500}
  ZCARD=3 不小于3 → 放行 → ZSet: {A:100, B:300, C:500}

时刻 t=600:  请求D来
  删除 score<600-1000=0（负数，无） → ZSet 不变
  ZCARD=3 不小于3 → 拒绝！（窗口内已有3个）

时刻 t=1150: 请求E来  ← 关键：窗口已经滑过A
  删除 score<1150-1000=150  → A（score=100）被删！→ ZSet: {B:300, C:500}
  ZCARD=2 < 3 → 放行 → ZSet: {B:300, C:500, E:1150}
```

注意 t=1150 这一刻：A 在 t=100 进来，到 t=1150 已经过去了 1050ms，超过窗口大小 1000ms。`ZREMRANGEBYSCORE` 把 A 删掉，窗口内容从 {A,B,C} 变成 {B,C}，腾出名额，E 得以放行。

窗口的"左边界"是 `now - 窗口大小`，随 now 动态变化；右边界是 now，每次请求来时更新。窗口不是匀速滑动，而是**每次请求来时瞬间对齐到当前时刻**。

**为什么不匀速滑动**：匀速滑动需要后台线程定时清理旧数据，有两个问题：①浪费 CPU（没请求时也空转清理）；②精度差（定时器有间隔，窗口边界不精确）。请求驱动方式：有请求才清理，没请求不动，既省资源又精确到毫秒。

**为什么 member 要用 UUID**：ZSet 里 member 必须唯一。如果两个请求同毫秒进来，member 都用时间戳的话，第二个会覆盖第一个，ZCARD 少算 1，限流不准。所以 member 用 UUID 或 时间戳+随机数 保证唯一，score 用时间戳保证排序。

**为什么不用令牌桶做用户级**：令牌桶要为每个用户维护一个桶状态（上次取令牌时间、剩余令牌数）。用户量千万级时，Redis 里要存千万个桶 key，内存爆炸。滑动窗口只存最近 1 秒内的请求，冷用户（1 秒内没动作的）通过 EXPIRE 自动清理，内存可控。活跃用户才占内存，符合二八定律。

#### 3. Sentinel：为什么用它熔断

**熔断在干什么**：类比电路保险丝——电流超载时自动跳闸，保护电器不被烧毁。软件熔断：下游（MySQL/Redis）变慢或报错达到阈值，自动"跳闸"一段时间，期间直接返回降级数据，不再调下游。

**为什么需要熔断（雪崩效应）**：

```
MySQL 变慢（查询从 10ms 变成 2s）
     ↓
跳转接口的线程都阻塞在等 MySQL
     ↓
Tomcat 线程池（默认 200）被占满
     ↓
新请求进不来，整个系统瘫痪
     ↓
连健康检查接口都响应不了，K8s 以为服务挂了开始重启
```

熔断就是：发现 MySQL 变慢，立刻停止调用它，返回本地缓存的老数据。虽然数据不是最新的，但系统能活着，用户能正常跳转。

**为什么用 Sentinel 而不是 Hystrix/Resilience4j**：

| 框架         | 状态            | 问题                                                         |
| ------------ | --------------- | ------------------------------------------------------------ |
| Hystrix      | 2018 年停止维护 | 不支持 SpringBoot 2.x 新特性，有已知 bug 不修                |
| Resilience4j | 活跃            | 函数式风格，与 Spring 集成需额外配置，学习曲线陡             |
| Sentinel     | 活跃维护        | 阿里开源，提供控制台可视化，与 Spring Cloud Alibaba 生态契合 |

Sentinel 还支持**慢调用比例熔断**（不只是异常比例）。数据库变慢但没报错的早期阶段，异常比例熔断抓不到（因为没异常），只有慢调用比例熔断能抓住。本项目两个策略都配了，双重保险。

**为什么降级返回 Caffeine 本地缓存**：

| 降级方案               | 问题                                      |
| ---------------------- | ----------------------------------------- |
| 返回错误页             | 用户体验差，短链点不开                    |
| 干等 MySQL 恢复        | 拖垮整个系统                              |
| 返回 Caffeine 本地缓存 | 0.05ms 返回，数据虽旧但能跳转，用户无感知 |

跳转接口的核心是返回长链 URL，即使缓存数据旧一点，跳转依然能成功。

#### 4. IP 黑名单：为什么自研

**为什么自研不用 Sentinel**：

| 维度 | Sentinel 限流              | IP 黑名单                    |
| ---- | -------------------------- | ---------------------------- |
| 语义 | "超出阈值拒绝当前请求"     | "进名单就全部拒绝，不管阈值" |
| 时效 | 实时判断，无"封禁时长"概念 | 封禁 1 小时，期间全部拒绝    |
| 粒度 | 基于 QPS/并发数            | 基于行为（持续高频）         |

Sentinel 的流控规则是"超出阈值就拒绝这一次"，但下一次请求来了又重新判断。黑名单是"一旦确认恶意，接下来 1 小时全部拒绝"，语义不同，Sentinel 做不到这种惩罚性封禁。

**为什么用分钟级而不是秒级**：秒级阈值（50 次/秒）对单用户够用，但对 IP 封禁太敏感——一个公司/校园网络出口 IP 共用一个，正常用户也可能短时高频（群里发了短链大家集体点），秒级 50 次很容易误判。分钟级（1000/min ≈ 16 次/秒）更宽松，只抓真正持续异常的刷量。

**为什么封禁 1 小时**：

- 太短（如 1 分钟）：惩罚不够，脚本换个时间继续刷
- 太长（如永久）：动态 IP 换人后误伤（运营商 NAT 池，同一 IP 过几小时可能换人）
- 1 小时：既能有效阻断刷量，又控制误伤窗口

## 深入理解：HyperLogLog 与内存聚合

> 这两个技术都服务于场景 C（访问统计），但解决的是完全不同的问题：HyperLogLog 解决"UV 去重太费内存"，内存聚合解决"海量事件不能来一条写一条"。

### 一、HyperLogLog：用 12KB 内存数 10 万个不重复的人

#### 解决什么问题

访问统计核心指标 UV（独立访客数）需要去重——10 万人点同一条短链，每人点 10 次，UV 应该是 10 万而不是 100 万。

三种去重方案对比：

| 方案                  | 10 万 UV 内存占用 | 问题                        |
| --------------------- | ----------------- | --------------------------- |
| MySQL DISTINCT        | 查时才算          | 全表扫描，慢到哭            |
| Redis Set             | ~1.5 MB           | 1 亿 UV 要 1.5 GB，内存爆炸 |
| **Redis HyperLogLog** | **固定 12 KB**    | 有 0.81% 误差               |

对 UV 这种"看个大概数量级就够了"的指标，0.81% 误差完全可接受（10 万 UV 误差 ±800 人），但省下的内存是数量级的。

#### 原理（核心反直觉点：它根本不存用户标识，只存一个数字）

**直觉理解——抛硬币**：让 10 万人每人抛硬币，记录每个人连续抛出正面的最高次数。连续抛出 k 个正面的概率是 `1/2^k`，所以观测到的最大连续次数 k，可以反推总人数大约是 `2^k`。

**实际实现——用哈希代替抛硬币**：

```
用户标识 → 哈希函数 → 一串 0 和 1（如 00010110...）
"连续正面" = "哈希值从左往右有几个连续的 0"
```

**为什么是 12KB——分桶降低误差**：单用一个 k 值估算误差太大（可能偏差 2 倍）。HyperLogLog 把哈希值的前 14 位用来分桶（2^14 = 16384 个桶），每个桶 6 bit 记录"落到本桶的元素中最大的连续 0 个数"：

```
16384 桶 × 6 bit = 98304 bit ≈ 12 KB  ← 固定不可变
```

单个桶的估算误差大，但 16384 个桶各自独立估算后用**调和平均**合并成一个总数，极端值被其他桶稀释，误差降到 0.81%。

**一句话总结**：取用户哈希值，前 14 位决定分到哪个桶，每个桶记录"落到本桶元素中最大的连续 0 个数"，所有桶用调和平均合并出总数，固定 12KB 内存估算任意规模的去重数量。

#### 本项目用法

```java
// 用户点击时，把用户标识扔进 HyperLogLog
redisTemplate.opsForHyperLogLog().add("uv:" + code, userId);
// 查询 UV 时直接读
Long uv = redisTemplate.opsForHyperLogLog().size("uv:" + code);
```

不是每次点击都 PFADD（太频繁），而是在内存聚合阶段批量 PFADD，和 60s 落库一起做。

---

### 二、消费者内存聚合：10 万次点击 → 几条统计结果

#### 解决什么问题

笨办法是来一条事件写一次库：1 秒 10 万次点击 = 10 万次 INSERT，数据库崩溃。

聚合的办法是先在内存里攒着、累加，定时批量写：10 万次点击 → 每 60 秒只产生 1 条 UPSERT。

**"聚合"就是把很多条原始记录，合并成少量统计结果，本质是 SQL 的 `GROUP BY + COUNT`，但放在内存里做。**

#### 聚合在算什么

假设 1 分钟内消费出 5 条事件：

```
事件1: {短码:abc, 时间:14:30, IP:1.1.1.1, 用户:user1}
事件2: {短码:abc, 时间:14:30, IP:1.1.1.1, 用户:user1}  ← 同一人点两次
事件3: {短码:abc, 时间:14:30, IP:2.2.2.2, 用户:user2}
事件4: {短码:xyz, 时间:14:30, IP:3.3.3.3, 用户:user3}
```

按「短码 + 日期 + 小时」分组算出：

```
短码 abc / 14点:  PV=3（不去重）  UV=2（user1+user2 去重）  IP数=2
短码 xyz / 14点:  PV=1  UV=1  IP数=1
```

#### 为什么在内存里做

| 做法                                 | 问题                                            |
| ------------------------------------ | ----------------------------------------------- |
| 直接写库（来一条写一条）             | 10 万次/秒，数据库崩                            |
| 查库聚合（攒一批再 SELECT GROUP BY） | 每次查库全表扫描，更慢                          |
| **内存聚合**                         | 10 万次累加在内存里毫秒完成，最后只 1 条 UPSERT |

#### 实现思路（两层 Map + 定时落库）

核心数据结构——两层 ConcurrentHashMap：

```
第一层: key = 短码
第二层: key = 日期+小时（如 "2026-07-09-14"）
值:    StatsBucket { pv: AtomicLong, uv: HyperLogLog, ips: Set<String>, ... }
```

工作流程：

```
Kafka 消费事件 → 从 Map 拿到对应桶 → 累加（pv+1, uv.add, ips.add）
                                        ↓
                        每 60 秒定时器触发
                                        ↓
遍历所有桶 → UPSERT 进数据库 → 清空桶，下周期从 0 重新累加
```

落库 SQL（利用唯一索引实现有则更新无则插入）：

```sql
INSERT INTO t_link_access_stats (short_code, stats_date, hour, pv, uv, ip_count)
VALUES ('abc', '2026-07-09', 14, 5000, 3000, 2500)
ON DUPLICATE KEY UPDATE
    pv = pv + VALUES(pv), uv = VALUES(uv), ip_count = VALUES(ip_count);
```

#### 线程安全

Kafka 消费者多线程并发消费，多个线程同时往同一个桶累加，必须线程安全：用 `ConcurrentHashMap`（分段锁/CAS）+ `AtomicLong`（无锁累加）+ HyperLogLog（Redis 单线程原子）。桶内部操作都是线程安全的，不需要额外加锁。

---

### 三、两者在场景 C 里的配合

```
用户点短链
   ↓
[本地队列] ← 削峰
   ↓
[Kafka] ← 缓冲
   ↓
[statsExecutor 线程池] ← 隔离：统计在自己的线程池里跑，不影响跳转
   │
   ├── 内存聚合（ConcurrentHashMap 累加 PV）  ← 省数据库压力
   ├── HyperLogLog 去重（算 UV，12KB 内存）   ← 省内存
   └── Redis Set 去重（算 IP 数，精确）
   ↓
[定时器 60s] ← 批量落库
```

- **HyperLogLog** 解决"UV 去重太费内存"的问题
- **内存聚合** 解决"海量事件不能来一条写一条"的问题

两个技术各管各的问题，共同保证访问统计功能既省资源又不拖垮主链路。

## 深入理解：分库分表

> 本节解决两个核心问题：① 项目里一共有哪些表、各自什么含义；② "分库分表"到底是什么意思、怎么分的。重点讲清"逻辑表"和"物理表"的关系——这是分库分表最容易混淆的概念。

### 一、一共有哪些表？各自什么含义

本项目共 **5 张逻辑表**，按数据特征分三类。

#### 1. t_short_link（短链主表）—— 核心业务表

存每一条短链的"档案"：短码、原始长链、创建者、过期时间、总 PV/UV、状态等。**数据量最大、查询最频繁**，是分库分表的主要对象。

#### 2. t_link_access_stats（访问统计表）—— 统计结果表

存聚合后的统计数据（不是每次点击一条，是按"短码+日期+小时"汇总）：PV、UV、IP 数、浏览器/地域分布。和短链表是"绑定关系"，数据量也大。

#### 3. t_link_access_record（访问明细表）—— 原始流水表

存每一次点击的原始记录（一条点击一条记录）：用户标识、IP、浏览器、操作系统、设备、地域、点击时间。**数据量爆炸级**（每次点击一条），按时间查询/归档为主。

#### 4. t_user（用户表）—— 广播表

存系统用户信息：用户名、密码、手机、邮箱。数据量小（几十到几百条）、几乎不变，但所有库都可能要查。

#### 5. t_group（分组表）—— 广播表

存短链分组（如"营销活动""个人收藏"）。同 t_user，数据量小、变动少。

#### 5 张表总览

| 表名 | 含义 | 数据量级 | 查询方式 | 分片策略 |
|------|------|---------|---------|---------|
| t_short_link | 短链主表 | 亿级 | 按 short_code 查 | 分库分表（哈希） |
| t_link_access_stats | 统计结果 | 千万级 | 按 short_code 查 | 分库分表（哈希）+ 绑定 |
| t_link_access_record | 访问明细 | 亿级/月 | 按时间查/归档 | 按月份分表 |
| t_user | 用户 | 几十~几百 | 全量查 | 广播表 |
| t_group | 分组 | 几十~几百 | 全量查 | 广播表 |

---

### 二、"分库分表"是什么意思

**分库** = 建多个数据库；**分表** = 每个数据库里建多张结构相同的表。

本项目的"4 库 × 4 表"意思是：建 4 个数据库（link_db_0~3），每个数据库里建 4 张结构相同的表（t_short_link_0~3），一共 4 × 4 = 16 张物理表。

---

### 三、核心概念：逻辑表 vs 物理表

这是分库分表最容易混淆的地方。

- **逻辑表**：应用代码里看到的表名，比如 `t_short_link`。你在 MyBatis 里写 `SELECT * FROM t_short_link`，看到的是一张表。
- **物理表**：数据库里真实存在的表，比如 `t_short_link_0`、`t_short_link_1`...这些才是真正存数据的地方。

**它们是"同一个表"，只是逻辑上是 1 张，物理上是多张。**

ShardingSphere 在中间做转换：

```
应用写：SELECT * FROM t_short_link WHERE short_code = 'abc'
                    ↓
            ShardingSphere 拦截
                    ↓
      算 hashCode：abc → 2
                    ↓
      改写为：SELECT * FROM t_short_link_2 WHERE short_code = 'abc'
                    ↓
            发给 ds_2 数据库执行
                    ↓
    应用以为查的是 t_short_link 一张表
    实际查的是 ds_2 库里的 t_short_link_2 这张物理表
```

应用永远只跟"逻辑表"打交道，不用关心数据存在哪张物理表。

#### 生活比喻

图书馆的目录系统显示"计算机类图书"（逻辑表），实际书分散在 4 个分馆的 4 个书架（物理表）。读者借书只说书名，管理员（ShardingSphere）帮你决定去哪个分馆哪个书架取。

### 四、5 张逻辑表各自变成多少张物理表

| 逻辑表 | 物理表 | 数量 |
|--------|--------|------|
| t_short_link | ds_0~3.t_short_link_0~3 | 4库 × 4表 = **16 张** |
| t_link_access_stats | ds_0~3.t_link_access_stats_0~3 | 4库 × 4表 = **16 张** |
| t_link_access_record | ds_0.t_link_access_record_202601~202612 | 1库 × 12月 = **12 张** |
| t_user | ds_0~3.t_user（内容相同） | 4库各1张 = **4 张** |
| t_group | ds_0~3.t_group（内容相同） | 4库各1张 = **4 张** |

### 五、四种分片策略详解

#### 策略 1：分库分表（t_short_link、t_link_access_stats）

**分片键**：short_code（短码）

**分库规则**：`ds_${short_code.hashCode().abs() % 4}`

**分表规则**：`t_short_link_${short_code.hashCode().abs() % 4}`

示例：
```
"abc" → hashCode=96354  → %4=2 → ds_2.t_short_link_2
"xyz" → hashCode=119429 → %4=1 → ds_1.t_short_link_1
```

**为什么用 short_code 做分片键**：短链跳转、详情查询都带 short_code，能精确定位到单分片。如果用 user_id，跳转时只有 short_code 没有 user_id，要查全部分片（广播查询），性能差。

**为什么分 4 库 × 4 表 = 16 片**：太少（如 2 片）扩容能力不足；太多（如 256 片）管理复杂。16 片让 1 亿条数据 → 单表 625 万，查询性能良好。

#### 策略 2：绑定表（t_short_link 和 t_link_access_stats）

**问题**：两张表都按 short_code 分片，如果不绑定，相同 short_code 可能落在不同库，join 时跨库。

```
不绑定：短链 abc 落在 ds_1，统计 abc 落在 ds_3 → 跨库 join，性能灾难
绑定后：短链 abc 和统计 abc 都落在 ds_1 → 同库 join，性能正常
```

绑定表保证两张表用相同分片算法，相同 short_code 一定落在同一个库。

#### 策略 3：按月份分表（t_link_access_record）

**分片键**：access_time（点击时间）

**分表规则**：按月份自动路由
```
access_time = 2026-07-09 → 月份 202607 → t_link_access_record_202607
access_time = 2026-08-15 → 月份 202608 → t_link_access_record_202608
```

**为什么按月份而不是按 short_code**：明细表主要按时间查询（"查最近一周的明细"），按时间归档方便（老数据直接 `DROP TABLE` 整张表）。只落 ds_0 不分库，因为明细表查询频率低，单库 + 月份分表已够用。

#### 策略 4：广播表（t_user、t_group）

每个库都存一份完整的相同数据。写入时 ShardingSphere 自动同步到所有库，查询时无论请求落到哪个库都能本地查到，不用跨库 join。

**为什么广播**：数据量小（几十条），每个库存一份不占空间；创建短链时要校验 user_id，无论短链落到哪个库都能直接 join 本库的 t_user。

### 七、雪花算法 + Base62 短码生成

**为什么需要**：分库分表后，MySQL 自增主键会冲突（4 个库各自从 1 自增，ID 重复）。

**雪花算法**：生成全局唯一 64 位 ID
```
[1位符号位][41位时间戳][10位机器ID][12位序列号]
     │          │           │           │
     不用     毫秒级时间   worker-id   毫秒内计数
```
- 41 位时间戳：可用 69 年
- 10 位机器 ID：支持 1024 台机器
- 12 位序列号：单机每毫秒可生成 4096 个 ID

**Base62 编码**：把雪花 ID 转成 6~7 位短码
```
雪花 ID = 1234567890123456789
       ↓ Base62 编码（0-9, A-Z, a-z 共 62 个字符）
短码 = "dnh8rT4"
```
62^6 ≈ 568 亿，足够用很久。含时钟回拨检测：发现当前时间 < 上次生成时间 → 报错或等待，保证 ID 唯一。

### 八、各类查询的路由方式

**按 short_code 查询（精确定位，单分片）**：
```sql
SELECT * FROM t_short_link WHERE short_code = 'aB3xK9';
```
→ 算 hashCode → 路由到 ds_3.t_short_link_3 → 单分片查询，快。

**不带分片键查询（广播查询，全分片）**：
```sql
SELECT * FROM t_short_link WHERE user_id = 1001;
```
→ 没有 short_code，ShardingSphere 不知道在哪个分片 → 发到所有 16 个分片 → 合并结果。代价大，本项目通过业务设计避免（先从 Redis 查缓存的 short_code 列表，再按 short_code 精确查）。

**按时间范围查询明细（月份路由）**：
```sql
SELECT * FROM t_link_access_record
WHERE access_time BETWEEN '2026-07-01' AND '2026-07-09';
```
→ 根据时间范围算出涉及 202607 这一张分表 → 只查这一张。

**绑定表 join（同库 join）**：
```sql
SELECT s.*, st.pv FROM t_short_link s
JOIN t_link_access_stats st ON s.short_code = st.short_code
WHERE s.short_code = 'aB3xK9';
```
→ 绑定表保证两张表相同 short_code 在同一库 → 同库 join，性能正常。
